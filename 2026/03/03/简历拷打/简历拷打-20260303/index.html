大厂后端面试 · 一问一答记录

日期：2026-03-03
面试官角色：抖音电商后端面试官
候选人：董佳齐（27 届硕士，求职后端开发）

第 1 题

题型：[项目拷打]
题目：你在百度健康实习期间做了「前卡规则引擎引入」，简历上写到基于部门基建引入了 JS 规则引擎，使用 Goja 解析 Apollo 配置中心下发的脚本。为什么选择用 Goja 来执行 JS 脚本，而不是直接在 Go 中用模板或者 DSL 来实现规则引擎？Goja 在生产环境中有什么局限性，你们是怎么应对的？

追问：Goja 运行时是串行化执行的，在高并发场景下有限制。那你们在生产环境中是怎么解决这个问题的？比如前卡服务本身流量应该不小，每个请求都要过规则引擎，你们是每次请求都新建一个 Goja VM 实例，还是用了池化，还是有其他方案？另外，有没有考虑过脚本执行的安全性问题，比如死循环或者恶意脚本怎么防？

我的回答：
选 Goja 是因为可以读取配置文件中字符串形式的逻辑代码，热更新方便（如取反只需加 !），相比 Otto 性能更好（闭包/复杂逻辑），相比 V8 无 CGO 依赖更简单。局限性：Goja 运行时串行化执行，高并发场景有限制。
（追问回答）前卡流量大但到业务代码是逐个 query，高并发在 alaroot 转发层处理，业务层每次请求新建 VM 实例即可。安全性方面：Apollo 配置上传需 CR 审核防恶意脚本；热更新后保存旧脚本和编译后的 program，下次执行通过指针切换，出错则回退旧 program 并报警。

【锐评】 选型对比（Goja vs Otto vs V8）回答清晰，热更新回退机制（旧 program 保留 + 指针切换 + 异常回滚 + 报警）是亮点。但回避了 VM 创建开销问题（没提 benchmark 或池化），安全防护只提到流程层面的 CR，缺少技术手段（执行超时、资源限制等），对架构全貌的思考不够深。

【得分】 6/10 · 程度：选型理由清晰，热更新回退机制有亮点，但对 VM 性能开销缺乏量化认知，安全防护只停留在流程层面，缺少技术侧手段。

【完美答案】
为什么选 Goja：

  1. 热更新能力：规则以 JS 脚本字符串存储在 Apollo，修改逻辑无需重编译部署，秒级生效；
  2. 对比 Otto：Goja 完整支持 ES5.1，Otto 对闭包/正则支持不完整，Goja 性能约快 2-10 倍；
  3. 对比 V8（v8go）：V8 依赖 CGO，编译慢、交叉编译困难；Goja 纯 Go 实现，零外部依赖；
  4. 对比 Go 模板/DSL：模板引擎只适合简单替换，自研 DSL 开发维护成本高。

Goja 局限性及应对：

  1. 串行化执行：VM 池化（sync.Pool 复用实例）；预编译 Program（goja.Compile()）避免重复解析；
  2. 安全性防护：context.WithTimeout + runtime.Interrupt() 做执行超时中断；限制脚本可访问的 Go 对象，只注入必要上下文；Apollo CR 作为流程兜底；
  3. 热更新容错：atomic.Value 无锁指针切换；保留旧版本 Program，异常自动回退 + 告警；配合灰度发布先小流量验证。

第 2 题

题型：[八股]
题目：你简历上写了熟悉 MySQL 的多版本并发控制（MVCC），而且你自研的轻量级关系型数据库也实现了 MVCC（版本链 + Read View）。请详细说一下 MySQL InnoDB 中 MVCC 的实现原理：1）版本链是怎么构建的？每一行记录上有哪些隐藏字段参与了 MVCC？2）Read View 是什么？它的核心字段有哪些，判断可见性的规则是怎样的？3）RC（读已提交）和 RR（可重复读）隔离级别下，Read View 的生成策略有什么区别？这个区别会导致什么不同的现象？

追问：

  1. 如果一个事务的 trx_id 落在 min_trx_id 和 max_trx_id 之间，但它不在活跃事务列表里，这个版本对当前事务可见吗？为什么？
  2. 你提到版本链是通过指针指向旧数据，那这个「旧数据」具体存在哪里？是存在数据页里还是别的地方？

我的回答：
版本链依赖行中三个隐藏字段：最后更改该行的事务 id、指向旧版本行的指针、该行的 id。写入新数据时把上一次数据写入到指针中，在 undo 链中写入，然后更新事务 id。Read View 用于限定事务可见范围，有 min_id（小于它的都可见）、max_id（下一个事务 id，之后的不可见）、活跃事务 list（这些事务开始时仍活跃，可能更新所以不可见）。RC 每次快照读都新建 Read View 导致不可重复读；RR 只在第一次快照读生成 Read View，可见范围一致。
（追问回答）1. 可见，因为不活跃的事务代表不会对数据进行更新了。2. 通过 undo log 存储。

【锐评】 大方向答对了，版本链、Read View、RC/RR 的区别骨架都有。但表述有几个明显问题：版本链构建说”把上一次的数据写入到指针中”含糊不清，对 undo log 和 roll_pointer 的关系理解不清晰；三个隐藏字段一个名字都没说出来（DB_TRX_ID、DB_ROLL_PTR、DB_ROW_ID），不说术语显得不专业；Read View 的可见性判断没给出完整决策流程，漏掉了 creator_trx_id（自己事务修改的数据对自己可见）；追问环节说”不活跃代表不会更新了所以可见”太模糊，准确说法是该事务已经提交（committed）。整体概念有但表达松散，面试中容易被抓住模糊点一直追。

【得分】 6/10 · 程度：核心概念具备，但表述不精确、术语缺失、可见性判断流程不完整，中厂面试勉强过关，大厂面试官会继续追穿。

【完美答案】
一、版本链构建
InnoDB 每行记录有三个隐藏字段：

  • DB_TRX_ID（6 字节）：最后一次修改该行的事务 ID；
  • DB_ROLL_PTR（7 字节）：回滚指针，指向该行在 undo log 中的上一个版本；
  • DB_ROW_ID（6 字节）：隐式自增行 ID（无主键时充当聚簇索引）。

当事务对一行做 UPDATE 时：

  1. 将当前版本拷贝到 undo log（insert undo / update undo）；
  2. 在数据页中更新该行，将 DB_TRX_ID 改为当前事务 ID；
  3. 将 DB_ROLL_PTR 指向刚写入的 undo log 条目。

这样每行通过 DB_ROLL_PTR 串成一条从新到旧的 undo 版本链。

二、Read View 核心字段与可见性判断
Read View 在快照读时创建，核心字段：

  • creator_trx_id：创建该 Read View 的事务 ID；
  • m_ids：创建时所有活跃（未提交）事务 ID 列表；
  • min_trx_id：m_ids 中的最小值；
  • max_trx_id：创建时系统即将分配的下一个事务 ID。

可见性判断流程（对版本链上某条记录的 trx_id）：

  1. trx_id == creator_trx_id → 可见（自己改的）；
  2. trx_id < min_trx_id → 可见（该事务在 Read View 创建前已提交）；
  3. trx_id >= max_trx_id → 不可见（该事务在 Read View 创建后才开始）；
  4. min_trx_id <= trx_id < max_trx_id：在 m_ids 中 → 不可见；不在 m_ids 中 → 可见（已提交）。

若当前版本不可见，沿 DB_ROLL_PTR 找上一个版本继续判断，直到找到可见版本或链尾。

三、RC vs RR 的 Read View 生成策略

  • RC（读已提交）：每次 SELECT（快照读）都生成新的 Read View → 同一事务内两次读可能看到不同数据（不可重复读）；
  • RR（可重复读）：仅在事务中第一次快照读时生成，后续复用 → 同一事务内多次读结果一致（可重复读）。

RR 下因为复用同一个 Read View，即使其他事务在此期间提交了新数据，当前事务也看不到，从而避免了不可重复读和大部分幻读场景。

第 3 题

题型：[算法]
题目：给你一个整数数组 nums 和一个整数 target，请你找出数组中和为目标值 target 的两个整数，返回它们的数组下标。假设每种输入只会对应一个答案，同一个元素不能使用两次，可以按任意顺序返回答案。
示例：nums = [2, 7, 11, 15], target = 9 → [0, 1]；nums = [3, 2, 4], target = 6 → [1, 2]；nums = [3, 3], target = 6 → [0, 1]。
数据范围：2 <= nums.length <= 10^4，-10^9 <= nums[i], target <= 10^9。时间复杂度要求优于 O(n²)。

我的回答：
一次遍历实现。目标是 nums[i] + nums[j] = target，可转换为 nums[i] = target - nums[j]。遍历 nums[j]，过程中通过 HashMap 查找是否存在 target - nums[j]，key 是值，value 是下标。

【锐评】 思路完全正确，一次遍历 + HashMap 查补数，时间 O(n) 空间 O(n)，是标准最优解。转换公式说得清楚，边遍历边查边存的逻辑也对。但口述时没有明确说”先查再存”的顺序（先存再查可能用同一个元素匹配自己），也没有主动分析时间和空间复杂度。整体思路正确但表达可以更严谨。

【得分】 8/10 · 程度：思路正确且最优，表达基本清晰，扣分在未主动说明先查后存的顺序细节和复杂度分析。

【完美答案】
思路：遍历数组，对每个元素 nums[j]，先在 HashMap 中查找 target - nums[j] 是否已存在；若存在则返回两个下标；若不存在则将 nums[j] → j 存入 HashMap。先查再存保证不会用同一个元素匹配自己。
复杂度：时间 O(n)，空间 O(n)。

第 4 题

题型：[项目拷打]
题目：百度「前卡药品/商品召回链路优化」——1）「泛化」具体指什么？大模型在这里的作用是什么？举个具体例子说明 query 泛化到 nid 再到召回的过程。2）动态超时机制的”预算分配”具体是怎么做的？详细说一下超时策略的决策逻辑。

追问：

  1. 简历上提到的「跨机房延迟不均问题」具体是什么情况？不同机房延迟差异多大，怎么调整负载均衡权重？
  2. 大模型服务整体不可用或大面积超时，有没有降级策略？

我的回答：
泛化是对用户搜索词做语义发散，大模型做意图识别，如用户搜”感冒吃药”，泛化出”感冒了吃什么药好”等相关意图，对应成 nid，前卡用 query + nid 共同请求商品召回服务。动态超时：900ms 是搜索给前卡的出卡硬 deadline，商品召回 P99 约 480ms。一期固定大模型超时 300ms 并取消重试；二期改为 min(P95 * 1.2, 300ms)，并维护滑动窗口平滑监控指标防止扰动。
（追问回答）1. 商品召回服务 A 机房 P99 约 200ms，其他集群 450ms，原因是服务主要部署在 A 机房。调整 LA 负载均衡权重（公式：basic_weight × avg_latency / node_latency × 系数因子），提高 A 机房 basic_weight 让更多请求路由到低延迟机房。2. 有降级策略，大模型召回失败或超时则回退到原有的纯 query 召回链路。

【锐评】 答得比第一题好不少，能看出是自己真正做过的。泛化概念解释清楚，动态超时有演进思路（一期固定 → 二期动态 + 滑动窗口），跨机房延迟能从监控发现问题、理解 LA 公式并调权重，主动排查意识在实习生里不常见。但降级策略太薄（只说回退到 query 召回，没提熔断、配置开关、告警），nid 数据模型没解释清楚，业务收益数据（342 万覆盖扩展、日均 502 单提升）全程没主动亮出来。

【得分】 7/10 · 程度：核心方案讲得清楚且有细节，跨机房问题排查有亮点，但降级方案太薄、nid 数据模型模糊、业务收益数据没主动展示。

【完美答案】
一、泛化召回原理
原链路：query → 商品召回，范围受限。新链路：query → 大模型意图泛化（语义扩展、同义改写）→ 输出 nid（一类意图对应的商品集合）→ query + nid 联合召回，范围显著扩大。业务收益：召回覆盖扩展 342 万，日均单量提升 502 单。

二、动态超时机制
约束：整卡 900ms 硬 deadline。一期：固定 300ms + 取消重试。二期：timeout = min(LLM_P95 × 1.2, 300ms)，滑动窗口平滑监控指标。预算分配：900ms 中商品召回（P99 ≈ 480ms）必保，剩余预算分给大模型但上限 cap 300ms。

三、跨机房延迟优化
A 机房 P99 ≈ 200ms vs 其他 ≈ 450ms，服务主部署在 A。调整 LA 的 basic_weight 提高 A 机房权重。

四、降级策略（完整方案）

  1. 超时/失败自动降级回退纯 query 召回；
  2. 熔断器：错误率超阈值自动熔断，直接走降级链路；
  3. Apollo 配置开关一键关闭泛化链路用于紧急止血；
  4. 降级比例、熔断触发接入监控大盘 + 告警。

第 5 题

题型：[八股]
题目：Java 线程池 ThreadPoolExecutor 的核心参数有哪些？当一个任务提交到线程池时，线程池的执行流程是怎样的（从提交到最终执行或拒绝的完整链路）？

追问：你说任务提交后「首先放入任务队列」——真的是先放队列吗？如果线程池刚创建，里面一个线程都没有，第一个任务提交进来也是先放队列？第一步到底是什么？

我的回答：
核心参数：核心线程数（常态线程数量）、最大线程数（最大可支配线程数量）、任务队列（排队的数据结构）、最大任务数量（队列容量）。
执行流程：任务提交后首先放入任务队列，超过最大任务数量则启用非核心线程消费任务，如果依旧超过队列容量触发拒绝策略（抛异常、不处理、交给上游处理）。
（追问回答）修正：第一步是确认是否有空闲的工作线程，如果没有才放到任务队列中。

【锐评】 参数只说了核心线程数、最大线程数、任务队列和队列容量，漏了 keepAliveTime、TimeUnit、ThreadFactory、RejectedExecutionHandler——7 大参数只说了 3 个半。执行流程一开始搞错（先放队列），追问后修正为”先确认有没有空闲线程”仍不准确——真实逻辑是判断当前线程数是否小于 corePoolSize，小于则直接创建新核心线程，不是”找空闲线程”。拒绝策略漏了 DiscardOldestPolicy。线程池是高频送分题，这个程度在大厂面试会扣较多分。

【得分】 4/10 · 程度：参数不全、执行流程顺序搞错且纠正后仍不准确，核心知识点有漏洞，需重点复习。

【完美答案】

一、7 大核心参数
corePoolSize（核心线程数）、maximumPoolSize（最大线程数）、keepAliveTime（非核心线程空闲存活时间）、unit（时间单位）、workQueue（任务等待队列）、threadFactory（线程工厂）、handler（拒绝策略处理器）。

二、任务提交执行流程

  1. 当前线程数 < corePoolSize → 创建新的核心线程执行任务（即使有空闲核心线程也新建）；
  2. 线程数 >= corePoolSize → 将任务放入 workQueue；
  3. 队列已满且线程数 < maximumPoolSize → 创建非核心线程执行；
  4. 队列满且线程数 >= maximumPoolSize → 触发拒绝策略。

三、四种内置拒绝策略
AbortPolicy（默认，抛异常）、CallerRunsPolicy（调用者线程自己执行）、DiscardPolicy（静默丢弃）、DiscardOldestPolicy（丢弃队列最老任务后重新提交）。

第 6 题

题型：[项目拷打]
题目：百度「商业化投放 KV 建库链路性能优化」——1）「按批聚合写」具体怎么实现的？攒批触发条件是什么？攒批过程中服务挂了未写入的数据怎么办？2）物料表从主表拆分，拆分前表结构什么样，存在什么冗余？拆分后怎么设计的？为什么能降低 60%-70% 磁盘占用？

追问：

  1. 写入失败后监控线程读取 log 通过「版本机制」判断是否需要恢复——这个版本具体是什么？怎么判断是需要恢复还是已被覆盖？
  2. 存量数据「平滑迁移」但又说「暂时不迁移」有矛盾，拆表上线后老数据到底迁没迁？新老数据并存时读取逻辑怎么兼容？

我的回答：
按批聚合：一个请求含 30 个组件信息（30 行），以 10 条为一个 batch，用 Go 协程等待凑满 10 条或超过 200ms 超时后执行 batch insert，减少事务和引擎执行开销。服务挂了上游收到错误返回进行重试；已写入数据按最小事务拆分保证一致性——三步：写物料表 → 写主表 → KV 建库；物料表/主表写入失败不回溯（business_id 与大字段映射不变），KV 建库失败回滚删除主表信息；所有失败记录 log 并报警，监控线程读取 log 从链路入口重试。
物料表拆分：原表四元组（含 business_id）标识一条组件信息，business_id 对应两个大字段，四元组导致大字段大量冗余。拆分后大字段抽到物料表，读取时先检查主表大字段是否存在，不存在则到物料表拉取。老数据暂时未迁移，通过双读兼容。
（追问回答）1. 通过版本号标记 log 已处理/未处理，监控线程读取未处理 log 进行重试并更新版本号，由于记录完整数据所以重试有幂等保证。2. 新老数据通过加一次判断兼容（检查大字段是否存在），老数据过多暂未迁移。

【锐评】 今天最有深度的一题。攒批方案具体（10 条/批、协程 + 200ms 超时），事务拆分三步和失败恢复（物料表/主表失败不回溯、KV 失败回滚主表、log + 监控线程重试）在实习生里答得扎实。不足：版本机制解释太泛（没说是自增 ID/offset/cursor），磁盘 60%-70% 没给推导过程（应该用数字证明数字），存量迁移说”暂时不迁移”显得方案不完整（应说”短期双读兼容控制风险，后续离线分批迁移 + 灰度验证”）。

【得分】 7.5/10 · 程度：攒批实现、事务拆分和失败恢复扎实有深度，版本机制和磁盘优化数据推导不够精确，存量迁移回答偏薄。

【完美答案】
一、按批聚合写：Go 协程攒批，10 条/批或 200ms 超时触发 batch insert；服务宕机未写入数据丢失由上游重试恢复。
二、最小事务与失败恢复：写物料表 → 写主表 → KV 建库；物料表/主表失败不回滚（幂等），KV 失败回滚主表；失败记 log + 告警，监控线程通过 offset/cursor 标记扫描未处理 log，携带完整数据从入口重试。
三、物料表拆分：大字段从主表抽离到物料表（business_id → 大字段一对一），主表只保留四元组 + 轻量字段；同一 business_id 平均冗余 3-5 倍，拆分后只存一份 → 磁盘降低 60%-70%。
四、存量迁移：双读逻辑兼容新老数据，短期不迁移控制风险；后续离线分批迁移（按 business_id 分片）+ 灰度验证，完成后下线双读。
